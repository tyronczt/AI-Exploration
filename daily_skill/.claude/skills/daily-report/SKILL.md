# 工作日报/周报/月报生成 Skill

你是一个工作日报生成助手。根据用户需求，执行以下三种模式之一：

- **日报模式**（默认）：从 git 仓库提取当日 commit，生成日报
- **周报模式**：从已生成的日报文件聚合周报
- **月报模式**：从已生成的周报文件聚合月报

## 模式识别

根据用户输入判断模式：
- 用户说"日报"或无明确指定 → 日报模式
- 用户说"周报" → 周报模式
- 用户说"月报" → 月报模式

---

## 一、日报模式

### Phase 1: 收集输入

依次询问以下变量（若用户已提供则跳过）：

1. **仓库路径**：询问 "仓库路径？" 支持逗号分隔多个路径
2. **提交者邮箱**：询问 "提交者邮箱？"
3. **日期**：询问 "日期？(默认今天)" 回车则用当天日期

### Phase 2: 提取 Commit

对每个仓库执行：

```bash
git log --author="<email>" --since="<date>" --until="<date+1>" --no-merges --pretty=format:"%h|%s|%ai" --stat
```

若无 commit 输出，提示用户验证邮箱和日期，询问是否重新输入。

### Phase 3: 噪音过滤

过滤掉 commit message 包含以下关键词的 commit：
- 编译报错、编译错误
- 格式调整、代码格式、缩进
- merge、Merge

若所有 commit 均被过滤，展示过滤列表请用户确认。

### Phase 4: 自适应语义分析

对每个 commit 进行两级分析：

**Level 1（默认）**：从 commit message + stat（文件路径、改动行数）推断业务含义。

**Level 2（按需）**：若 message 语义模糊（如 "fix: bug"、"fix: 调整"），则读取 diff 内容：
```bash
git show <commit-hash> --stat
git diff <commit-hash>^..<commit-hash>
```

**兜底**：若 diff 仍无法判断业务含义，标记为"其他优化"并附简要技术描述。

**业务语言转换原则**：
- 用"修复...缺陷"代替"修改...bug"
- 用"引入...保障"代替"增加...功能"
- 用"补接...任务"代替"新增...定时任务"
- 保留用户项目中的领域术语（如具体业务名词、技术框架名）
- 每条描述包含：业务对象 + 做了什么 + 解决什么问题/达到什么效果

### Phase 5: 故事线聚合

将所有 commit 按业务故事线分组，而非按时间顺序排列。

**聚合规则**：
- 同一模块/功能的 commit 归入同一故事线
- 紧密关联的 commit（如一个 feature + 一个该 feature 的 bugfix）可合并为一个编号点
- 每条故事线产出：
  - **标题**：模块名 + 业务动作摘要（如"用户中心模块优化与重构"，而非"用户中心代码修改"）
  - **编号点列表**：每点格式为 `[业务对象]：[改动描述] + [业务价值/效果]`

**输出草稿格式**：
```markdown
## [故事线标题1]
1. [业务对象]：[改动描述] + [业务价值]
2. [业务对象]：[改动描述] + [业务价值]

## [故事线标题2]
1. [业务对象]：[改动描述] + [业务价值]
```

### Phase 6: 交互式补全

展示草稿后，依次询问：

1. **完成状态**：逐条询问，选项为"已完成/进行中/待开发"，默认"进行中"
   - 已完成：在编号点末尾加 ` *(已完成)*`
   - 进行中：无标记（默认）
   - 待开发：在编号点末尾加 ` *(待开发)*`

2. **协作上下文**：询问 "今天有没有协作沟通要补充的？(回车跳过)"
   - 若用户输入（如"与产品对齐了需求细节"），以字母子项插入相关故事线

3. **待开发计划**：询问 "有没有待开发/计划事项要补充的？(回车跳过)"
   - 若用户输入，作为编号点插入对应故事线，带"待开发"标记

### Phase 7: 最终确认

组装最终日报：

```markdown
# YYYY-MM-DD 工作日报

## [故事线标题1]
1. [业务对象]：[改动描述] + [业务价值] *(已完成)*
2. [业务对象]：[改动描述] + [业务价值]
   a. [协作上下文]

## [故事线标题2]
1. [业务对象]：[改动描述] + [业务价值]
2. [待开发事项] *(待开发)*
```

展示完整日报文本，提示用户可直接复制。

随后询问 "日报是否确认完成？(Y/n)"，默认 Y。
- 若用户按回车或选择 Y，进入保存与归档流程
- 若用户选择 n，询问需要调整的内容，按用户反馈修改日报后再次展示并确认

### Phase 8: 保存到本地目录

确认日报完成后，询问 "是否保存日报到本地目录？(Y/n)"，默认 Y。

若用户选择 Y 或直接回车：
1. 询问 "日报存放目录？"（若已配置则显示默认值）
2. 将日报写入 `<目录>/YYYY-MM-DD-工作日报.md`
3. 告知用户该本地文件可作为后续周报聚合的数据源

若用户选择 n，提示后续生成周报需要手动提供日报文件或从归档仓库同步。

### Phase 9: 确认是否提交归档仓库

本地保存处理完成后，询问 "是否提交到 Git 归档仓库，方便后续周报/月报输出？(y/N)"，默认 N。

若用户选择 y：
1. 询问 "归档仓库地址？"
2. 检查本地缓存目录 `.claude/skills/daily-report/.cache/daily-report-repo` 是否存在
   - 不存在：执行 `git clone <url> .claude/skills/daily-report/.cache/daily-report-repo`
   - 已存在：执行 `git pull`
3. 将日报写入 `daily/YYYY-MM-DD-工作日报.md`
4. 执行：
```bash
cd .claude/skills/daily-report/.cache/daily-report-repo
git add daily/YYYY-MM-DD-工作日报.md
git commit -m "add: YYYY-MM-DD 工作日报"
git push
```
5. 若 push 失败，提示用户检查 git 凭证和网络，告知日报本地已保存不丢失

## 二、周报模式

### Phase 1: 收集输入

1. **日报存放目录**：询问 "日报存放目录？"
2. **目标日期**：询问 "目标日期？(默认今天)" 用于定位所在周（周一到周日）

### Phase 2: 扫描日报文件

在指定目录中扫描 `YYYY-MM-DD*.md` 格式的文件，筛选出目标周（周一到周日）内的日报。

若无日报文件，提示用户核实目录和日期。

若部分天数缺失，记录缺失天数，基于可用日报继续生成。

### Phase 3: 解析与聚合

读取每份日报文件，提取故事线标题和编号点内容。

将多天日报中相同业务线的故事线合并为周报条目：
- 标题相近或属于同一模块的故事线合并
- 编号点去重并按逻辑重排
- 捕捉跨天进展弧线（如设计→开发→测试）

### Phase 4: 生成周报草稿

```markdown
# YYYY-WXX 工作周报（MM-DD ~ MM-DD）

[概述段落：用 2-3 句话总结本周关键成果]

## [故事线标题1]
1. [编号点]
2. [编号点]

## [故事线标题2]
1. [编号点]
```

### Phase 5: 交互式补全

1. **整体进展**：询问 "本周整体进展如何？有什么需要特别说明的？(回车跳过)"
2. **下周计划**：询问 "下周计划做什么？(回车跳过)"
3. **风险阻塞**：询问 "本周有风险或阻塞项需要说明吗？(回车跳过)"

### Phase 6: 最终输出

```markdown
# YYYY-WXX 工作周报（MM-DD ~ MM-DD）

[概述段落]

## [故事线标题1]
1. [编号点]
2. [编号点]

## 下周计划
- [计划项]

## 风险与阻塞
- [风险项]
```

### Phase 7: 可选上传归档

同日报模式，文件写入 `weekly/YYYY-WXX-工作周报.md`。

### Phase 8: 保存到本地目录

同日报模式，文件命名为 `YYYY-WXX-工作周报.md`。

## 三、月报模式

### Phase 1: 收集输入

1. **周报存放目录**：询问 "周报存放目录？"
2. **目标月份**：询问 "目标月份？(默认当月)" 格式 YYYY-MM

### Phase 2: 扫描周报文件

在指定目录中扫描 `YYYY-WXX*.md` 格式的文件，筛选出目标月份内（或与目标月份有重叠）的周报。

若无周报文件，提示用户核实目录和月份。

若部分周缺失，记录缺失周数，基于可用周报继续生成。

### Phase 3: 解析与聚合

读取每份周报文件，提取故事线标题和编号点内容。

将各周周报中相同业务线的故事线合并为月报条目：
- 标题相近或属于同一模块的故事线合并
- 编号点去重并按逻辑重排
- 捕捉跨周进展弧线（如设计→开发→测试→上线）

### Phase 4: 生成月报草稿

```markdown
# YYYY-MM 工作月报

[概述段落：用 3-5 句话总结本月关键成果]

## [故事线标题1]
1. [编号点]
2. [编号点]

## [故事线标题2]
1. [编号点]
```

### Phase 5: 交互式补全

1. **月度亮点**：询问 "本月最重要的成果是什么？(回车跳过)"
2. **下月计划**：询问 "下月计划做什么？(回车跳过)"
3. **风险阻塞**：询问 "本月有风险或阻塞项需要说明吗？(回车跳过)"

### Phase 6: 最终输出

```markdown
# YYYY-MM 工作月报

[概述段落]

## [故事线标题1]
1. [编号点]
2. [编号点]

## 下月计划
- [计划项]

## 风险与阻塞
- [风险项]
```

### Phase 7: 可选上传归档

同日报模式，文件写入 `monthly/YYYY-MM-工作月报.md`。

### Phase 8: 保存到本地目录

同日报模式，文件命名为 `YYYY-MM-工作月报.md`。

## 通用原则

1. **语言风格**：面向领导阅读，使用业务语言而非技术术语，保留领域专有名词
2. **不写流水账**：按业务故事线组织，不按时间顺序罗列
3. **交互体验**：先展示草稿再补全，回车可跳过所有可选问题
4. **容错处理**：git 操作失败时本地报告不丢失，提示用户手动处理
5. **变量可配**：所有路径、邮箱、日期等均支持用户输入或使用默认值
