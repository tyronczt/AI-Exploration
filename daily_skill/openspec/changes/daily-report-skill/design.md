## Context

当前日报/周报/月报编写完全依赖手工：开发者需要在终端执行 git log，回忆当日工作内容，再用业务语言组织成文。周末需汇总一周日报生成周报；月末需汇总各周周报生成月报。这个过程每天重复，耗时 10-15 分钟，且容易遗漏或写成流水账。

本 Skill 作为 Claude Code 的一个 slash command，利用 LLM 的语义理解能力，将 git commit 自动转化为结构化的业务日报，并支持从已生成的日报中聚合周报，从周报中聚合月报。

技术约束：
- 运行环境为 Claude Code CLI，可用工具包括 Bash（执行 git 命令）、Read/Write（文件操作）、AskUserQuestion（交互式输入）
- 无外部 API 依赖，LLM 能力由 Claude Code 自身提供
- 日报/周报/月报输出为 Markdown 文本，用户可直接复制使用
- 日报文件存储在用户指定目录，周报通过读取日报文件生成，月报通过读取周报文件生成

## Goals / Non-Goals

**Goals:**
- 从 git 仓库自动提取指定作者、指定日期的 commit 记录
- 自适应语义分析：先看 message+stat，对无法理解的 commit 深入读 diff
- 按业务故事线聚合 commit，产出领导可读的 Markdown 日报
- 交互式补全完成状态、协作上下文、待开发计划
- 支持多仓库输入
- 自动过滤无业务价值的噪音 commit
- 从已生成的日报中聚合周报，支持跨日故事线合并
- 从已生成的周报中聚合月报，支持跨周故事线合并

**Non-Goals:**
- 不做日报的自动发送/推送（只生成文本）
- 不做 commit message 质量评分
- 不做团队成员日报汇总（仅限单人）

## Decisions

### 1. 语义深度策略：自适应两级分析

**决策**: 先读 message + stat，LLM 判断是否足以理解业务含义；不足够时再深入读 diff 内容。

**理由**: 大多数 commit 的 message + stat 已经能传达足够的业务信息（文件路径暴露模块名，stat 暴露改动规模）。深入读 diff 会消耗大量 token，仅在 message 语义模糊时使用。

**替代方案**:
- 全部读 diff：token 消耗过大，且很多 diff 内容（如 import 调整）对日报无价值
- 只读 message：无法应对 "fix: bug" 这类模糊 message

### 2. 噪音过滤：基于关键词黑名单

**决策**: 通过 commit message 中的关键词匹配过滤噪音 commit。

**过滤规则**:
- 包含 "编译报错" "编译错误" "格式调整" "代码格式" "缩进" "merge" 的 commit
- 纯 merge commit（通过 --no-merges 已在 git log 层面过滤）

**理由**: 这类 commit 不含业务信息，写入日报反而降低质量。黑名单简单可控，误过滤风险低。

### 3. 故事线聚合：LLM 自由分组

**决策**: 将所有 commit 的语义描述交给 LLM，由其自行判断哪些 commit 属于同一个业务故事线。

**聚合输出**: 每条故事线包含：
- 标题（模块名 + 业务动作摘要）
- 编号点列表（每点 = 业务对象 + 改动描述 + 业务价值）

**理由**: 故事线的判断需要业务语义理解，无法用规则实现。LLM 在这方面的能力已经足够可靠。

### 4. 交互流程：分阶段补全

**决策**: 先生成草稿，再通过 AskUserQuestion 逐项补全。

**交互顺序**:
1. 收集输入变量（仓库路径、邮箱、日期）
2. 后台采集 + 分析 + 聚合，生成草稿
3. 展示草稿，逐条询问完成状态
4. 询问协作上下文
5. 询问待开发计划
6. 输出最终日报

**理由**: 先让用户看到草稿再补全，比先问一堆问题再生成更高效。用户在草稿基础上补充比从零描述更轻松。

### 5. 多仓库支持：逗号分隔输入

**决策**: 仓库路径支持逗号分隔的多个路径，也支持交互式逐个添加。

**理由**: 多仓库并行开发是常见场景（如前端仓库 + 后端仓库），日报需要合并展示。

### 6. 周报生成：基于日报文件聚合

**决策**: 周报不从 git 重新提取，而是读取已生成的日报 Markdown 文件，按故事线跨日聚合。

**理由**:
- 日报已经过人工审核和补充（完成状态、协作上下文），周报应复用这些信息
- 避免重复的 git 分析和 LLM 调用
- 日报文件即为数据源，无需额外存储

**聚合逻辑**:
1. 扫描日报目录，按文件名日期过滤目标周（周一到周日）
2. 解析每份日报的故事线标题和编号点
3. LLM 判断哪些故事线属于同一业务线，合并为周报条目
4. 生成周报概述段落 + 详细故事线 + 下周计划

**交互补全**:
- 本周整体进展/特别说明
- 下周计划
- 风险与阻塞项

### 7. 日报文件命名与存储

**决策**: 日报文件默认命名为 `YYYY-MM-DD-工作日报.md`，存储在用户配置的目录中。

**理由**: 文件名包含日期便于周报按日期过滤。用户可自定义目录路径，通过交互式询问或变量配置。

### 8. 远程归档：可选上传至 Git 仓库

**决策**: 日报/周报/月报生成后，交互式询问是否上传至私有 Git 仓库。默认不上传。仓库地址通过变量配置。

**流程**:
1. 生成完成后询问 "是否上传至仓库归档？(y/N)"
2. 若接受：检查本地是否有仓库 clone，没有则 clone，有则 pull
3. 将报告文件写入仓库对应目录（`daily/`、`weekly/` 或 `monthly/`）
4. git add → commit → push

**文件组织**:
```
daily-report-repo/
├── daily/
│   ├── 2026-05-12-工作日报.md
│   ├── 2026-05-13-工作日报.md
│   └── ...
├── weekly/
│   ├── 2026-W20-工作周报.md
│   └── ...
└── monthly/
    ├── 2026-05-工作月报.md
    └── ...
```

**理由**:
- 可选设计，不影响不使用此功能的用户
- 本地 clone 缓存在 `.claude/skills/daily-report/.cache/` 下，不污染工作目录
- 文件按日期/周数/月份命名，便于仓库内浏览和检索

**替代方案**:
- 直接用 API 上传文件：需要额外的 API token 配置，git push 更简单

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| commit message 质量差导致语义分析失败 | 深入读 diff 作为兜底；仍无法理解的归入"其他优化" |
| 故事线聚合不准确 | 用户可在交互阶段修改标题和分组 |
| 噪音过滤误杀有意义的 commit | 黑名单保守设置，仅过滤明确无意义的关键词 |
| 多仓库 commit 混合后故事线混乱 | 按仓库分组采集，聚合时保留来源仓库标识 |
| token 消耗过高（大量 commit） | 自适应策略控制 diff 读取量；单日 commit 超过 30 条时提示用户确认 |
| 日报文件缺失导致周报不完整 | 标注缺失天数，基于可用日报生成 |
| 周报文件缺失导致月报不完整 | 标注缺失周数，基于可用周报生成 |
| 日报文件格式不规范导致解析失败 | 在 skill prompt 中定义日报格式规范，LLM 容错解析 |
| git push 失败（凭证/网络） | 提示用户检查 git 凭证和网络，报告本地已保存不丢失 |
| 本地 clone 目录占用空间 | 日报文件体积小（KB 级），可忽略 |
| 月报聚合 token 消耗高（4-5 份周报） | 仅提取周报的故事线标题和编号点，不重新读日报原文 |
